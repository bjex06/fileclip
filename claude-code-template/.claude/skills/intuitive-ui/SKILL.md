# Intuitive UI Design

## 設計思想

> **「ITリテラシーが低くても使えて、使ううちにリテラシーが上がる」**

これは**ガードレールを最小限にする**スキル。判断の余地を大きく残す。

## 3つの問い

UI設計時に自問する：

1. **説明なしで操作できるか？** → No なら改善
2. **間違えにくいか？** → No なら確認ステップ追加
3. **使うほど理解が深まるか？** → No なら段階的開示

## パターン集

具体的な実装パターンは状況に応じて選択：

→ ./patterns/progressive-disclosure.md（段階的開示）
→ ./patterns/contextual-help.md（ツールチップ・ヘルプ）
→ ./patterns/confirmation.md（危険操作の確認）
→ ./patterns/feedback.md（操作フィードバック）

## 判断ガイド

| 状況 | 適用パターン |
|------|-------------|
| 初回ユーザー | オンボーディング |
| 機能が多い | 段階的開示 |
| 専門用語がある | ツールチップ |
| 取り消せない操作 | 確認ダイアログ |
| 処理に時間かかる | ローディング + 完了通知 |

## やらないこと

- 全機能を最初から見せる
- 専門用語を説明なしで使う
- 「本当に削除しますか？」だけの確認（何が消えるか書く）
- ローディング中に何も表示しない

## 参考：悪い例 → 良い例

```
❌ 削除しますか？ [はい] [いいえ]
✅ 「プロジェクトA」を削除しますか？この操作は取り消せません。 [キャンセル] [削除する]

❌ エラーが発生しました
✅ メールアドレスの形式が正しくありません（例: name@example.com）

❌ 設定 > 詳細設定 > システム > 通知 > メール通知
✅ 基本設定画面に「メール通知」トグル
```
